# Combined Recommendations — Coverage Grid Launch Page

> Generated by Discovery Phases 1–7
> Date: 2026-03-03
> Companion files: `agent-roster.md`, `DISCOVERY-LOG.md`

---

## Executive Summary

Replace the tarva-launch capsule ring (6 hardcoded districts in a circular layout) with a coverage-category card grid backed by live TarvaRI intel data. The spatial ZUI engine, ambient effects, morph choreography, header/footer, and navigation HUD all stay. The old ring becomes a CSS Grid of coverage category cards. Clicking a card triggers the existing morph drill-down into a data-driven district view.

**Scope:** ~16 new files, ~18 modified, ~14 archived. 8 work areas across 4 execution phases. Project is approximately 2–3x larger than a surface read of the goals suggests.

---

## Technical Decisions (Recommended)

### 1. Type Strategy: Generic NodeId

Replace `DistrictId` (6-member union) with `NodeId = string`. Create a new `CategoryMeta` interface for coverage-specific metadata. The morph system, UI store, and spatial actions all accept any string ID — no need for two parallel type systems.

**Why:** The morph state machine (`MorphState.targetId`), choreography hook, and detail panel are already generic in logic — only the type annotation constrains them to 6 IDs. Widening to `string` unlocks dynamic categories with minimal code change.

### 2. Grid Layout: 1500×400px at World Origin

CSS Grid container with `grid-template-columns: repeat(8, 1fr)`. Each card ~160×180px with 16px gaps. Container centered at world origin (0, 0).

**Why:** Fits between existing ambient panels (SystemStatusPanel at x:-1200, FeedPanel at x:+880) with ~130px clearance on each side. Two rows handle up to 16 categories.

### 3. Morph Animation: Scale + Fade

Selected card scales to 1.2x, siblings fade to opacity 0.3, detail panel slides in from the right. Reuse existing morph phase timing (expanding: 400ms, settling: 200ms).

**Why:** Simpler than ring rotation, visually clear, and the existing `morph-panels-scatter` CSS already handles pushing data panels outward during morph. The morph phase state machine stays unchanged.

### 4. Category List: Hybrid (Known Set + Dynamic Counts)

Define `KNOWN_CATEGORIES` constant with 15 categories from the data spec (seismic, geological, disaster, humanitarian, health, aviation, maritime, infrastructure, weather, conflict, fire, flood, storm, multi-hazard, other). Each entry has display name, color, and icon. Query `intel_sources` for actual source counts. Only show cards for categories with ≥1 source.

**Why:** Gives stable color/icon mapping while adapting to whatever data exists. Unknown categories from DB fall back to "other" styling.

### 5. Map Library: MapLibre GL JS

Use `maplibre-gl` + `react-map-gl` (MapLibre backend). Free, no token, vector tiles, WebGL rendering, ~200KB gzipped.

**Critical placement decision:** The map should live inside the **district view overlay** (which is `position: fixed; inset: 0;` — not CSS-transformed) rather than in world-space. WebGL canvases do not render correctly inside CSS `transform: scale()` containers. This means the map appears only when drilling into a category, not on the launch page overview.

**Why:** MapLibre is the standard open-source choice. Placing it in the district view avoids the WebGL-in-CSS-transforms problem entirely.

### 6. District View: Single Generic Scene

One `CategoryDetailScene` component that accepts a category ID and renders:
- Filtered alert list (from `intel_normalized` for that category)
- Severity breakdown (bar chart or ring)
- Source health table (from `intel_sources` for that category)
- Category-filtered map (MapLibre, zoomed to that category's geographic footprint)

**Why:** 15+ categories make per-category hand-crafted scenes impractical. Data-driven approach means new categories automatically work.

### 7. Filter State: Separate Zustand Store

New `coverage.store.ts` with:
- `selectedCategory: string | null`
- `setSelectedCategory(id: string)`
- `clearSelection()`

URL sync via `?category={id}` replacing the old `?district={id}`.

**Why:** Keeps data filtering separate from morph animation state. The morph system drives visual transitions; the coverage store drives data queries. They sync when a card click triggers both.

## Risk Mitigation

| Risk | Severity | Mitigation |
|------|----------|------------|
| DistrictId type cascade breaks build | HIGH | Do WA1 first, gate with `pnpm typecheck` before proceeding |
| WebGL map inside CSS transforms | HIGH | Place map in district view overlay only (fixed position, no transforms) |
| Morph orchestrator rewrite breaks animations | HIGH | Keep morph phase state machine unchanged, only change visual transition |
| Supabase tables missing or schema mismatch | MEDIUM | Build hooks with graceful empty/error states first, verify against live Supabase |
| @tarva/ui Card doesn't achieve glass effect | LOW | Card + `bg-black/70 backdrop-blur-sm` confirmed in exploration |
| Grid of 8 too wide at some zoom levels | LOW | CSS Grid is responsive by nature; cards can wrap. Test at Z0 through Z2. |
